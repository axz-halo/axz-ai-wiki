슬래시 커맨드

슬래시 커맨드는 자주 쓰는 워크플로우를 한 단어로 줄이는 사용자 정의 명령입니다. /review · /security-review 같은 기본 명령과 사내 전용 /deploy-check까지 직접 만들 수 있습니다.

이 챕터의 목차
  1. 기본 슬래시 커맨드
  2. /review 출력 해석법
  3. 사내 슬래시 만들기
  4. 매개변수가 있는 슬래시
  5. 슬래시 체이닝
  6. 사내 슬래시 카탈로그

기본 슬래시 커맨드

Claude Code의 기본 패키지에 포함된 핵심 슬래시 3개입니다. 처음엔 이 셋만으로도 체감이 옵니다.

명령용도쓰는 순간
/initCLAUDE.md 자동 작성새 프로젝트 인식
/review코드 / PR / diff 리뷰리뷰 PR 다듬기 전
/security-review보안 리뷰 · 시크릿 점검 · 입력 검증배포 전
팁 · 리뷰는 두 번
PR 다듬기엔 /review, 보안 리뷰엔 /security-review를 따로 돌리세요. 두 명령은 보는 관점이 완전히 다릅니다.

/review 출력 해석법

/review는 결과를 blocker / suggestion / nit 세 단계로 분류합니다. 보안 점검 severity 패턴이 동일하고, suggestion은 너그럽게 보되 처음엔 모든 nit을 다 보세요.

BLOCKER · 반드시 수정 · 보안·기능 결함. 보고 의무.

SUGGEST · 가능하면 수정 · 가독성·성능·디자인 이슈.

NIT · 취향 · 시간 여유 시.

사내 슬래시 만들기

.claude/commands/ 디렉터리에 Markdown 파일을 두면 그 슬래시가 프로젝트 전용 명령입니다. 파일 이름이 곧 슬래시 이름. 파일 본문이 프롬프트 템플릿입니다.

# .claude/commands/deploy-check.md
---
description: 배포 직전 운영 PR을 점검·검증·요약
---

다음 체크리스트를 차근차근 수행하고 다음 단어로 답변.
1. `pnpm typecheck` 0 에러
2. `pnpm test` 전체 통과
3. `pnpm build` 성공
4. .env.example과 .env.production 키 일치
5. 새로 추가된 API 라우트는 `auth-middleware` 통과
   결과는 ✓/✗ 표 형식으로 답해.

이제 세션에서 /deploy-check만 입력하면 이 템플릿이 실행됩니다. 보고된 점검 결과로 누락된 점검 사항이 한 페이지에 정리됩니다.

주의 · 비밀 키는 슬래시에 넣지 마세요
슬래시 파일은 Git에 커밋됩니다. API 키·DB 비밀번호 같은 건 환경 변수만 쓰고, 템플릿엔 "환경 변수 SLACK_TOKEN을 읽음"만 적습니다.

슬래시 파일 구조

.claude/
├── commands/                  # 슬래시 커맨드
│   ├── deploy-check.md
│   ├── retro.md
│   └── review.md
├── settings.local.json        # 로컬 환경 변수 (gitignore)
├── scripts/                   # 환경 스크립트
│   └── pre-commit-claude.sh
└── settings.shared.json       # 팀 공유 설정 (commit)

매개변수가 있는 슬래시

Markdown 본문에 $ARGUMENTS 자리표시자를 두면 슬래시 뒤 인자가 자동 치환됩니다. 단순 인자, 다중 인자 둘 다 가능합니다.

# .claude/commands/ticket.md
---
description: Linear 인자 요청 템플릿
---

Linear 인자 $ARGUMENTS 를 읽어 다음 형식으로 정리해.
- 영향 범위:
- 우선 순위:
- 의존성 인자:
- AC (수락 기준):
- 액션 아이템:
> /ticket AXZ-1243

인자가 여러 개인 경우 $1, $2를 쓰거나 자연어 "첫 번째 인자" / "두 번째 인자"로 작성합니다.

# .claude/commands/release-note.md
---
description: Changelog 자동 작성 (버전 $1과 $2)
---

# Changelog $1 ... $2

버전 사이의 커밋을 다음 형식으로 정리.
## ✨ New
## 🐛 Fixes
## ⚡ Changes
## ⚠ Breaking

슬래시 체이닝

하나의 슬래시 안에서 여러 슬래시를 순차로 호출하면 한 단어로 여러 작업을 묶을 수 있습니다.

# .claude/commands/ship.md
---
description: 배포 직전 리뷰·보안·체크 라이프사이클 자동화
---

다음 작업을 순서로 해줘.
1. /review를 먼저 → blocker가 0이면 다음
2. /security-review를 먼저 → high가 0이면 다음
3. /deploy-check를 먼저 → 9개 미통과 없음
4. 통과되면 "SHIP READY · 9개" 출력
   하나라도 미통과면 미통과 인자만 표시
한 줄
/ship 한 글자가 리뷰·보안·배포 체크를 한 번에 돌립니다. 반복 작업은 모두 이렇게 줄이세요.

슬래시 디버깅

✓ 파일이 .claude/commands/에 있는가

✓ Frontmatter에 ---로 닫힌 description이 있는가

✓ 슬래시 이름에 공백·한글이 없는가

✓ 인자 자리표시자 정확히 $ARGUMENTS 또는 $1로 표기됐는가

✓ /를 치면 미리보기 목차에 떴는가

사내 슬래시 카탈로그 소개

커뮤니티팀이 관리하는 사내 표준 25개의 슬래시 번들입니다. 처음엔 이 4개만 익히세요.

/deploy-check · 배포 직전 9개 체크. 플랫폼팀.

/ticket · Linear 인자 자동 요약 + 정리. PMO.

/a11y · 접근성 체크. 디자인시스템팀.

/sql-review · SQL 쿼리 리뷰·튜닝. DBA.

새 슬래시 제안하기

커뮤니티 팀이 관리하는 슬래시 axz-corp/claude-commands 리포지터리에 PR을 던지세요. 커뮤니티 팀 리뷰 후 사내 표준으로 등록됩니다. PR엔 반드시 사용 시나리오 1건과 예상 호출 빈도를 명시합니다.

CASE · 디자인시스템팀

/tokens 슬래시

배경
Figma 토큰 변경 후 코드 동기화 일감이 1주에 며칠씩 차지.
슬래시
Figma MCP로 토큰 새로 받기 + tailwind.config.ts 자동 변경 + diff 리뷰.
결과
토큰 동기화 시간 1주 → 1h. 사내 표준 등록.

이 챕터가 끝나면

✓ /review·/security-review 둘 다 돌려 봤다.

✓ 사내 슬래시를 1개 직접 만들어 봤다.

✓ 인자 자리표시자($ARGUMENTS)와 다중 인자($1·$2)를 구분한다.

✓ 슬래시 체이닝으로 여러 작업을 한 단어로 묶었다.
