한 명의 Claude가 모든 일을 다 하면 컨텍스트가 빠르게 더러워집니다. 서브에이전트로 역할을 분담하고, 훅으로 트리거를 자동화하면 Claude Code의 실력이 두 배가 됩니다.

서브에이전트 세 가지 패턴

다른 Claude를 별도로 호출해 격리된 컨텍스트에서 일을 시키는 방식입니다. 세 가지 패턴이 가장 자주 쓰입니다.

01 Plan — 구현 전 설계

큰 PR을 시작하기 전에 Plan 서브에이전트에게 먼저 설계를 맡깁니다. 메인 컨텍스트를 복잡하게 만들지 않고 여러 안을 검토할 수 있습니다.

02 Explore — 넓은 탐색

"이 기능이 어디서 다 쓰이지?" 같은 광범위한 grep·glob 결과는 토큰이 폭증합니다. 서브에이전트가 결과만 추려서 가져옵니다.

03 Test Runner — 루프 자동화

"테스트 통과할 때까지 고쳐줘"를 서브에이전트로 돌리면 메인이 시끄럽지 않게 끝까지 갑니다. 디프만 메인으로 보고됩니다.

팁 · 병렬 호출
독립적인 일은 한 번에 여러 서브에이전트를 동시 호출하세요. parallel·isolated · agent·call을 함께 쓰면 체감 속도가 배가 됩니다.

컨텍스트 분리 도식

FIGURE 2-3 · 메인 컨텍스트 vs 서브에이전트 컨텍스트

서브에이전트는 격리된 컨텍스트에서 작업하고, 결과만 메인으로 돌려보냅니다. 메인의 토큰·집중도가 보존됩니다.

훅으로 트리거 자동화

훅은 특정 이벤트가 일어날 때 자동으로 실행되는 스크립트입니다. .claude/hooks.json에 등록하면 Claude가 도구를 호출할 때마다 검사·로깅이 됩니다.

{
  "hooks": {
    "pre-tool-use": [{
      "matcher": { "tool_name": "Write" },
      "command": "bash .claude/scripts/check-secrets.sh"
    }],
    "post-tool-use": [{
      "matcher": { "tool_name": "Bash" },
      "command": "bash .claude/scripts/log-bash.sh"
    }]
  }
}

훅 이벤트 종류

이벤트발동 시점활용
session-start세션 시작 시 1회기본 컨텍스트 주입, 로그 시작
pre-tool-use도구 실행 직전시크릿·금지 명령 차단

pre-commit 훅 연동

Git의 pre-commit과 Claude Code의 훅을 연결하면 커밋 직전 Claude가 자동으로 리뷰합니다. Husky를 쓰는 팀에서 가장 손쉽습니다.

#!/bin/sh
# .husky/pre-commit
claude review --staged --non-interactive --severity blocker
exit_code=$?

if [ $exit_code -ne 0 ]; then
  echo "Claude 리뷰에서 blocker 발견. 수정 후 다시 커밋."
  exit 1
fi
비용·속도 균형
모든 커밋에 리뷰를 걸면 CI 비용이 큽니다. 사내 권장은 pre-push 시점 1회. 자잘한 커밋엔 안 걸고, 푸시 직전에만 한 번 검사.

실전 케이스 4종

CASE 1 · FE팀

Playwright 테스트 깃발 뽑기

상황
378개 테스트, 깜빡임(flaky) 8.2%. test-stabilizer 서브에이전트에게 "한 줄 보고서 20개씩 보내고, 단계별 수정"을 맡김.
결과
5주 후 단계별 적용 25 PR 머지, 90일 안정성 92→98%. 메인 컨텍스트는 깨끗하게 유지.
CASE 2 · 백엔드

대규모 리팩터 — 커넥터 교체

상황
120 라이브러리에서 legacy SDK 호출을 신규 SDK로 일괄 치환. refactor-runner 서브에이전트가 파일별 변환 + 라이브러리 테스트 검증.
결과
8일 동안 실패 박스 45개 → 0. 서브 컨텍스트가 diff만 메인에 보고.
CASE 3 · 데이터

SQL 쿼리 성능 튜닝

상황
느린 쿼리 TOP 10을 sql-optimizer 서브에이전트에 넘김. 쿼리별 인덱스·실행 계획 비교·재작성 자동 생성.
결과
평균 12,200ms → 1,400ms. 매 분기 8.7배 빨라진 시점.
CASE 4 · QA

릴리스 노트 3개 국어 작성

상황
Changelog 초안을 병렬로 3명 서브에이전트(KO·EN·JA)에 던짐.
결과
총 12분. 병렬 3건. 일관성은 메인 컨텍스트에서 검수.

훅 스크립트 테스트법

훅은 실행되어야 의미가 있으므로, 검증 전 짧은 시뮬레이션 루틴이 필요합니다. 세 단계를 거치세요.

exit 1 대신 echo 'WARN'으로 시작하세요. 1주일간 로그만 모으고 패턴이 명확해지면 차단으로 전환합니다.

이 챕터가 끝나면

✓ Plan · Explore · Test Runner 세 패턴 중 어떤 일에 무엇을 쓸지 30초 안에 결정한다.

✓ 사내 레포에 pre-commit 훅 하나는 직접 등록해 봤다.

✓ .env·credentials 같은 시크릿 파일이 커밋에 못 들어가도록 훅이 차단한다.
