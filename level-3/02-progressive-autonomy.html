점진적 자율성

에이전트가 신뢰를 얻는 속도만큼 권한을 키웁니다. Lv.1 제안형 → Lv.2 검토 후 실행 → Lv.3 백그라운드 자동 실행의 3단계를 승인률·쿨다운 데이터 기반으로 승·강등합니다.

자율성 게이트 로직

승급은 자동이 아닙니다. 데이터 기반 추천 + 사용자 확인의 2단계를 거칩니다. 세 가지 규칙을 함께 두세요.

01 · 승급 기준 — 20회 중 17회 이상 승인

동일 액션 유형 기준 롤링 윈도우 20회 중 17회 이상 승인이면 다음 레벨을 추천합니다. 추천일 뿐, 확정은 사용자 다이얼로그를 거쳐야 합니다.

02 · 쿨다운 — 마지막 강등 이후 7일

Critical Incident로 강등되면 7일간 재승급 제안을 차단합니다. 신뢰 리빌드는 점진적이어야 합니다.

03 · 액션 영역별 독립 추적

"파일 수정"과 "외부 API 호출"은 별개 윈도우로. 한 영역의 신뢰가 다른 영역으로 자동 이월되지 않습니다.

# autonomy_gate.py
def recommend_next_level(action_type: str, user_id: str) -> int | None:
    hist = fetch_recent_actions(user_id, action_type, limit=20)
    if len(hist) < 20: return None
    approval_rate = sum(h.approved for h in hist) / len(hist)
    if approval_rate < 0.85: return None
    if days_since_last_demotion(user_id) < 7: return None
    return min(current_level(user_id) + 1, 3)

3레벨 토글 UI

사용자는 언제든 레벨을 수동 조정할 수 있어야 합니다. 버튼 자체가 현재 에이전트 상태(실행 중/대기/중단)를 컬러로 드러냅니다.

// AutonomyToggle.tsx
export function AutonomyToggle({ level, onChange }: Props) {
  const labels = ["제안만", "검토 후 실행", "자동 실행"];
  return (
    <div className="flex gap-1 rounded-lg bg-stone-100 p-1">
      {labels.map((label, i) => (
        <button
          key={i}
          onClick={() => onChange(i + 1)}
          className={level === i + 1
            ? "bg-orange-500 text-white px-3 py-1.5 rounded"
            : "text-stone-600 px-3 py-1.5"}
        >
          Lv.{i + 1} · {label}
        </button>
      ))}
    </div>
  );
}
레벨 버튼은 헤더에 둘 것
설정 메뉴 안에 깊이 숨기면 사용자는 자율성을 통제한다는 감각을 잃습니다. 헤더 우측 같은 잘 보이는 자리에 둬야 안심하고 권한을 올립니다.

승급 UX — 좋은 예 / 나쁜 예

BAD

"새로운 기능 사용 가능!" 같은 팝업을 배너로 띄움.

권한 상승이라는 인식 없이 버튼을 누르게 함. 사용자가 왜 레벨이 올랐는지 모름.

GOOD

"지난 20번 중 18번을 승인하셨어요. 이제 자동 실행으로 올릴까요? 언제든 Lv.1로 되돌릴 수 있습니다."

수치와 되돌리기 옵션을 함께 제시. 사용자가 통제하고 있다는 감각이 유지됨.

팁 · 14일 침묵 옵션
다이얼로그 하단에 "이 결정은 14일간 다시 묻지 않음" 체크박스를 두면 숙련 사용자의 클릭 피로가 줄어듭니다. 단 14일 뒤에는 반드시 리프레시.

강등 규칙

자동 강등은 사람 검토 전 단계에서 에이전트 피해를 멈추는 안전핀입니다. 3가지 트리거를 YAML로 선언합니다.

# downgrade_rules.yaml
triggers:
  - name: critical_incident
    when: action.outcome == "data_loss" OR action.outcome == "prod_break"
    action: demote_to(1)
    notify: ["#alert-agent", "owner"]

  - name: consecutive_rejections
    when: rolling_window(5).reject_count >= 3
    action: demote_to(level - 1)
    cooldown_days: 7

  - name: cost_spike
    when: hourly_cost_usd > budget.hourly * 3
    action: demote_to(1)
    pause_minutes: 30
절대 하지 말 것
강등을 사용자에게 알리지 않는 것. 레벨이 바뀌었으면 즉시 토스트 + 슬랙 알림 + 대시보드 배지를 띄워 "왜 에이전트가 갑자기 소극적이 됐지?" 의문을 30초 안에 해소해야 합니다.

실전 강등 케이스 3종

CASE 1 · 데이터 사고

Prod DB UPDATE — data_loss 트리거

발생
한 Lv.3 에이전트가 스키마 변경 PR을 자동 적용하며 UPDATE ... WITHOUT WHERE를 실행. 5만 행 덮어씀.
대응
규칙이 즉시 Lv.1 강제 + 14일 쿨다운 + 포스트모템 의무화. 같은 사용자의 자동 승급 제안은 14일 후에도 2주간 승인률 90%+로 제한.
학습
"WHERE 없는 UPDATE 금지" 가드레일을 pre-tool-use 훅에 추가.
CASE 2 · 비용 사고

Cost Spike — 시간당 $420 도달

발생
AWS 분석 스크립트가 재귀 호출로 무한 루프.
대응
규칙이 30분 일시 중지 + Lv.1 강제 + 플랫폼팀 알림. 비용은 $420에서 멈춤.
학습
사후 분석 결과 Claude가 pagination 파라미터를 잘못 이해해 반복 호출. 프롬프트 템플릿에 "pagination 명시 필수" 가드레일 추가.
CASE 3 · 사용자 피드백 사고

Consecutive Rejections — 5번 중 3번 거부

발생
신규 사용자가 동료의 권한을 옮겨받아 Lv.3 상태로 시작. 5번 중 3번 "이건 아니에요" 취소.
대응
규칙이 한 단계 강등(Lv.3 → Lv.2)하고 7일 쿨다운.
학습
사용자 인터뷰에서 "통제감 회복됐다"는 피드백. 권한은 양도 불가 항목으로 변경.

자율성 안티패턴 5선

안티패턴왜 위험한가
"전사 Lv.3" 강제전사 공지로 강제 승급. 신뢰 축적 없이 실패 폭주.
조용한 강등알림 없이 강등하면 "에이전트 망가짐"으로 오해 유발.
액션 영역 통합 추적한 영역 신뢰가 타 영역으로 이월되어 사고 확산.
강등 직후 재승급 제안강등 다음 날 재승급 제안. 신뢰 리빌드 기회 놓침.
되돌리기 버튼 숨기기사용자가 Lv.1로 내릴 버튼을 숨김. 통제감 상실.

이 챕터가 끝나면

✓ 사내 에이전트 한 개에 Lv.1/2/3 토글이 헤더에 노출되어 있다.

✓ downgrade_rules.yaml 3종 트리거(critical_incident · consecutive_rejections · cost_spike)가 운영되고 있다.

✓ 강등 발생 시 토스트 + 슬랙 알림이 30초 안에 사용자에게 도달한다.

✓ 액션 영역별로 별도 윈도우로 추적되어 있다.

자율성을 단단히 잡았다면, 이어 3-3 컨텍스트 보존으로 세션 단절을 메우러 갑니다.
