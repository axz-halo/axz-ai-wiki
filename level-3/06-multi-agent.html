싱글 vs 멀티 결정 기준

싱글 에이전트로 충분

단일 도메인 · 컨텍스트 20k 토큰 이내 · 순차 처리 · 한 사람이 모든 결정.

멀티 필요

이질적 도메인 병렬 · 컨텍스트 > 50k 토큰 · 역할 분리 필수 · 다수의 독립 책임자.

결정의 원칙
의심스럽다면 싱글로 시작하세요. 싱글로 풀 수 있는 일을 멀티로 풀면 디버깅 비용이 3~5배로 늘어납니다.

Opus 4.7 · 에이전트 티어 라우팅

2026년 3월 출시된 Claude Opus 4.7은 다단계 추론과 tool_use 정확도가 크게 향상되어, Planner 단계에 적합한 모델입니다. 고급편에서는 비용 대비 성능 곡선을 고려해 3개 모델을 티어로 나눠 쓰는 Tiered Routing을 표준으로 삼습니다.

FIGURE 3-6a · 3-TIER ROUTING (PLANNER / EXECUTOR / CHECKER)
from anthropic import Anthropic
from typing import Literal

Tier = Literal["planner", "executor", "checker"]
MODEL_BY_TIER = {
    "planner":  "claude-opus-4-7",
    "executor": "claude-sonnet-4-6",
    "checker":  "claude-haiku-4-5-20251001",
}

class AgentRouter:
    def __init__(self, budget_opus_calls: int = 8):
        self.client = Anthropic()
        self.opus_left = budget_opus_calls

    def call(self, tier: Tier, messages, tools=None):
        if tier == "planner" and self.opus_left <= 0:
            tier = "executor"   # fallback
        if tier == "planner":
            self.opus_left -= 1
        return self.client.messages.create(
            model=MODEL_BY_TIER[tier],
            max_tokens=4096,
            messages=messages,
            tools=tools or [],
        )
비용 통제
4.7은 4.6과 가격이 같지만 Sonnet 대비 약 5배. budget_opus_calls 세션당 8회가 기본. 초과 시 자동으로 Sonnet fallback. 월간 리뷰에서 Opus 실행 평균 횟수를 1번 지표로 추적하세요.

오케스트레이션 3 패턴 — 개요

세 가지 패턴은 문제의 성격에 따라 섞어 쓸 수도 있습니다. A는 조직의 팀장-팀원 구조, B는 연구팀의 병행 실험, C는 공장의 조립 라인을 닮았습니다.

패턴구조비유
A · Supervisor한 명이 감독 · 직렬 분배팀장-팀원
B · Swarm병렬 탐색 · 투표/합의연구팀 병행 실험
C · Pipeline고정 스테이지 직렬공장 조립 라인

패턴 A · Supervisor

한 에이전트(Supervisor)가 하위 전문 에이전트에게 작업을 순차로 위임합니다. 의사결정 책임이 한 곳에 모이므로 디버깅이 가장 쉽습니다.

# supervisor.py
class Supervisor:
    def __init__(self):
        self.workers = {
            "research": researcher_agent,
            "code":     coder_agent,
            "review":   reviewer_agent,
        }
    def run(self, task):
        plan = self.plan(task)             # 단계 분해
        for step in plan:
            worker = self.workers[step.role]
            result = worker.execute(step)
            if not self.ok(result):
                return self.retry_or_fail(step, result)
        return self.synthesize(plan)
장점디버깅 쉬움, 책임 명확
단점직렬 → 총 지연 시간 큼
용도코드 리뷰 봇 · 릴리즈 오케스트레이터

패턴 B · Swarm

N개의 에이전트가 같은 문제를 병렬로 풀고, 결과를 투표·합의로 모읍니다. 정확도가 중요한 분류·요약에 유리합니다.

# swarm.py
async def swarm_answer(question, n=5):
    responses = await asyncio.gather(*[
        agent.ask(question) for _ in range(n)
    ])
    counter = Counter(r.answer for r in responses)
    top, freq = counter.most_common(1)[0]
    if freq > n // 2:
        return top
    return max(responses, key=lambda r: r.confidence).answer
장점견고성·정확도 상승
단점비용 N배, 결과 해석 복잡
용도컴플라이언스 라벨링 · 요약 합의
비용 경고
Swarm은 토큰 비용이 N배입니다. Opus를 다수로 돌리지 마세요. Haiku 5병렬이 Opus 1회보다 나은 경우가 대부분.

패턴 C · Pipeline

고정된 파이프라인을 따르는 에이전트 조립 라인입니다. 각 단계가 다음 단계의 입력을 생성합니다. 반복 실행이 많은 배치 작업에 적합합니다.

# pipeline.py
class Pipeline:
    stages = [extractor, normalizer, enricher, validator, writer]
    def run(self, doc):
        ctx = {"input": doc}
        for stage in self.stages:
            ctx = stage.process(ctx)
            if ctx.get("error"):
                return self.rollback(ctx)
        return ctx["output"]
장점재현성·관측성·재시작성 최고
단점유연성 낮음, 설계 비용 큼
용도문서 파이프라인 · 데이터 ETL · OCR 후처리

CASE 2 · SWARM

Compliance Labeler — 5병렬 Haiku 다수결

법무 문구 라벨링에 Haiku 4.5를 5병렬로 돌려 다수결. 단일 Haiku 정확도 86% → 5병렬 투표 94%. 비용은 5배였으나 사람 재검토 시간이 2/3 감소해 순비용 오히려 하락.

CASE 3 · PIPELINE

지출 영수증 OCR 자동화 — 5스테이지 패션 정리

extract(PDF→text) → normalize(단위/통화) → enrich(카테고리) → validate(스키마+중복) → write(DB). 5스테이지 각각 독립 테스트, 실패 시 이전 스테이지로 자동 롤백.

주 3만건 처리, 실패율 0.7%.

메시지 버스와 상태 공유

3 패턴 중 어느 것을 선택하든, 에이전트 간 공유 상태를 관리할 계층이 필요합니다. 사내 기본은 NATS(경량 메시지 버스) + Redis(상태 스냅샷) 조합입니다.

규칙
에이전트 간 공유는 메시지로만. 다른 에이전트의 내부 메모리를 직접 읽지 말 것. 격리가 곧 복구력입니다.

장애 시 격리 원칙 — 3 규칙

1. Bulkhead 분리 · 한 에이전트 실패가 전체 정지로 번지지 않게. 타임아웃 + 회로차단기 필수.

2. Idempotency · 재시작 가능하려면 같은 입력에 같은 결과. 부작용 있는 단계에는 idempotency key.

3. Dead Letter Queue · 실패한 작업은 조용히 버리지 말고 DLQ에 보존, 알림 발생.
