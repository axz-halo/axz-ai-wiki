기획자·디자이너의 Claude Code 자율 사용

플랫폼팀이 사전 세팅·슬래시·MCP·안전 가드를 잘 깔아두면, 기획자와 디자이너가 터미널을 열어 코드를 모르고도 Claude Code를 자유자재로 쓸 수 있습니다. 이 장은 "비개발 직군이 Claude Code를 쓰게 만드는" 플랫폼 책임자를 위한 챕터입니다.

왜 "비개발자도 Claude Code"인가

기획자·디자이너의 Claude Code 직접 사용은 단순 편의가 아닙니다. 조직 차원에서 3가지 구조적 이점이 있습니다.

01 요구 번역 비용 제거 · PM이 스펙을 말로 설명 → 개발자가 코드로 번역 → 의도 왜곡, 라는 만성 루프가 사라집니다. PM이 직접 Claude Code로 실제 파일을 만지기 때문입니다.

02 디자인·코드 왕복 단축 · 디자이너가 컴포넌트 수정을 자연어로 지시하고 PR까지 올림. 엔지니어는 리뷰에만 집중. 사이클 타임 4~6배.

03 엔지니어 인지 부하 분산 · 반복적·자명한 변경은 PM·디자이너가 처리, 엔지니어는 아키텍처·성능·보안에 집중. 팀 처리량 구조 자체가 변합니다.

짚어둘 점
"비개발자가 Claude Code를 쓴다"는 개발자 대체가 아닙니다. 반복 작업을 분산시켜 팀 처리량을 올리는 구조 설계입니다. 엔지니어의 코드 리뷰는 여전히 필수 게이트입니다.

플랫폼 사전 세팅 7단계

비개발자가 두려움 없이 Claude Code를 쓰려면 플랫폼팀이 먼저 깔아야 할 7가지가 있습니다. 순서를 지켜 설정하세요.

#단계설명
01샌드박스 리포실수해도 사내 운영에 영향 없는 격리 레포
02CLAUDE.md 프리셋역할·금지·체크리스트 사전 주입
03슬래시 팩PM/디자이너 전용 5~7개 명령
04MCP 커넥터Notion·Figma·Slack 읽기 전용
05보안 훅Ch2-6 5대 사례 기반 방어막

# CLAUDE.md — 비개발자용 프리셋

## 이 사람은 누구?
기획자/디자이너. 코드 규칙을 직접 모름.
모든 응답은 "왜 이 변경이 필요한지" 한 줄 먼저 설명.

## 기본 규칙
- 새로 만드는 파일은 항상 Git 브랜치에서만. main 직접 금지.
- node_modules, .env, dist/ 수정 절대 금지.
- "테스트 돌려봐" 요청 시 자동으로 npm test 실행.
- 에러가 2회 반복되면 즉시 중단하고 사람에게 질문.

## 언어
답변은 한국어. 코드 주석도 한국어.

## 자동 에스컬레이션
- 파일 10개 이상 변경 필요 시 → 엔지니어 리뷰 요청
- 마이그레이션·DB 변경 → 즉시 중단
- 외부 API 신규 호출 → 보안팀 승인 요청

PM 시나리오 · "스펙이 코드가 되는 하루"

PM이 화요일 오전에 Claude Code 세션을 열어 금요일 배포까지 가는 실제 흐름. 단계마다 어떤 슬래시를 쓰는지, 엔지니어는 어디서 개입하는지를 시간 축으로 보여줍니다.

핵심
PM은 코드를 쓰지 않지만 무엇을 원하는지를 정확히 표현해 Claude Code에게 작업을 위임합니다. 엔지니어 개입은 "리뷰" 단 한 번으로 줄어듭니다.

디자이너 시나리오 · 시안 → 코드 직송

디자이너가 Figma에서 수정한 시안을 Claude Code에게 "이대로 코드로 반영해"라고 말하면 PR까지 자동 생성됩니다. 핵심은 디자인 토큰 동기화입니다.

# 1) Figma에서 수정 완료 후 공유 링크 복사
# 2) Claude Code 세션에서
/design handoff https://figma.com/file/ABC123 \
  --framework react \
  --tokens ./design-tokens.json \
  --scope components/Button,components/Card

# 3) Claude 출력
✓ Figma 변경 8개 감지
✓ 6개는 토큰 업데이트로 해결 (재빌드 불필요)
✓ 2개는 컴포넌트 코드 변경 필요
  - Button: padding 12/16 → 14/20
  - Card: border-radius 8 → 12
→ PR 초안 생성: design-sync-2026-04-19

안전 장치 3종

1. 디자인 시스템 외 색 차단 · 토큰 외 색 hex를 쓰려 하면 훅이 차단. "시스템 색 중 가장 가까운 것으로 치환할까요?" 묻도록 설계.

2. 로직 파일 수정 금지 · src/components/**와 토큰 파일만 허용. src/api/·src/store/ 변경 시 훅이 즉시 중단 후 엔지니어 호출.

3. 접근성 회귀 자동 감지 · 색 대비·ARIA 속성 변경이 WCAG AA 기준에 미달하면 PR 생성 전 자동 실패.

안전 가드 · 비개발자가 뚫을 수 없는 5대 벽

비개발자 Claude Code 운영의 성패는 무엇을 못 하게 막을지가 결정합니다. 5대 벽은 모두 플랫폼팀이 일회 설정으로 구축합니다.

1. 브랜치 보호 · main 직접 푸시 불가. PR만 허용. CODEOWNERS에 엔지니어 최소 1명 필수.

2. 허용 경로 화이트리스트 · 비개발자 세션은 허용 경로 외 쓰기 자동 차단. .claude/allowed-paths.json에 선언.

3. 의존성 잠금 · package.json, pyproject.toml 등 의존성 파일 변경은 엔지니어 승인 필요.

4. 비용 한도 · 비개발자 세션당 Opus 4.7 호출 8회·Sonnet 200회 초과 시 자동 종료. 하루 누적 한도도 별도.

5. 실 로그 보관 · 모든 tool_use·파일 수정·PR 생성이 6개월 OpenTelemetry에 저장. 분기별 감사 리뷰 필수.

// .claude/allowed-paths.json
{
  "role": "pm-designer",
  "allow_write": [
    "src/components/**",
    "src/pages/**",
    "i18n/**/*.json",
    "design-tokens.json",
    "docs/specs/**"
  ],
  "deny_write": [
    "src/api/**",
    "src/store/**",
    "migrations/**",
    "**/*.env*",
    "infra/**"
  ],
  "require_engineer_review": [
    "package.json",
    "pyproject.toml",
    ".github/workflows/**"
  ]
}
주의 · 초기 6주는 학습 모드
벽이 너무 높으면 오히려 팀 생산성이 떨어집니다. 초기 6주는 로그만 수집하고 실제 차단은 하지 마세요. 7주차부터 "반복 차단 감지" 경로만 실제 차단으로 전환. 가이드가 아니라 학습한 규칙이 되어야 합니다.

학습 로드맵 · 비개발자 4주 커리큘럼

플랫폼팀이 PM·디자이너를 온보딩할 때 쓰는 4주 커리큘럼. 각 주차는 3시간 세미나 + 자가 실습 숙제로 구성됩니다.

주차주제실습 슬래시숙제
W1Claude Code 기본 · 샌드박스 첫 PR/page new샌드박스에 "내 소개" 페이지 1개 만들어 PR 머지
W2카피·토큰 수정 · 디자인 핸드오프/copy edit · /design handoffUX 카피 10건 수정, 토큰 2개 변경 PR
W3실무 리포에서 작은 변경/component tweak실제 제품 컴포넌트 2개 수정, 엔지니어 리뷰 통과
W4플로우·상태 · PR 자가 리뷰/flow fix · /ship draft상태 추가 요청 1건을 처음부터 끝까지 본인이 완료

달성 등급 · 비개발자 Claude Code 4단계

등급설명
L0 · 관찰엔지니어 세션 참관, 자기 브랜치 없음
L1 · 모방샌드박스 리포에서만 단독 작업
L2 · 기여실 리포에 PR, 엔지니어 리뷰 거침
L3 · 단독작은 기능을 처음부터 끝까지 리드
팁 · 속도보다 통과율
L0 → L3 도달 평균 기간은 약 8주. 조급해하지 말고 주당 2건 이하의 PR을 목표로. 속도보다 "엔지니어 리뷰 pass율 80%"가 현실적 기준입니다.

이 챕터가 끝나면

✓ 비개발자 전용 CLAUDE.md 프리셋이 사내 표준에 등록됐다.

✓ 5종 슬래시 팩을 PM·디자이너에게 제공할 준비가 끝났다.

✓ allowed-paths.json 5대 벽이 설치됐다.

✓ 4주 커리큘럼 캘린더를 잡았다.
