PART 1 · Ralph 자가 루프 개념 (20분)

1.1 "Ralph"란 무엇인가

"Ralph"는 2025년 개발자 커뮤니티에서 유행한 극단적으로 단순한 자가 반복 에이전트 루프의 별명입니다. 복잡한 오케스트레이션 없이 "한 파일을 한 프롬프트로 계속 개선하게 두는" 접근입니다.

# Ralph 루프 · 최소 의사 코드
while True:
    claude.run(
      "AGENTS.md를 읽고, 다음 한 가지 할 일을 선택해 구현하라. "
      "완료되면 상태를 AGENTS.md에 기록하고 커밋하라.",
      tools=[Read, Edit, Bash, Glob]
    )
    if is_done(AGENTS.md):
        break

1.2 Ralph가 주는 교훈 3가지

01 단순함이 가장 강한 프로토콜이다 · 수십 개 에이전트를 서로 엮는 복잡한 설계보다, "한 파일을 끝까지 물고 늘어지는 루프"가 더 멀리 간다. 복잡도는 실패 지점을 늘릴 뿐.

02 상태는 파일에 산다 · 메모리·세션이 아니라 AGENTS.md 같은 텍스트 파일에 상태를 누적하면 중단·재개가 손쉽다. 고급 본책 3-3 HANDOFF.md와 같은 철학.

03 도구는 적게, 경계는 명확히 · Read·Edit·Bash 같은 저수준 도구 몇 개면 충분. 위험한 쓰기는 훅으로 차단. 이 조합이 실제 산출물 품질을 가장 안정적으로 올린다.

Ralph는 "프로토타입 철학"이다
Ralph 스타일은 해커톤·1인 개발에 최적화돼 있습니다. 다인 팀·프로덕션 운영에는 L3 본책 Ch6~7의 Supervisor·Pipeline + 하네스가 더 안전합니다. 개념은 흡수하되, 그대로 프로덕션에 꽂지 마세요.

1.3 하네스 엔지니어링 · 3세대 지형도

"하네스(Harness)"는 에이전트의 입출력·실행·관측·복구를 감싸는 실행 환경입니다. 2024~2026년 사이 지형은 세 세대를 거쳤습니다.

3세대 하네스의 필수 6요소

✓ Trace 스트림 — tool.call · tool.result 이벤트가 타임라인으로 보임

✓ 정책 훅 — pre-tool-use로 위험 호출을 사전에 막음

✓ 비용 게이지 — 세션·일간 토큰·호출 비용 실시간

✓ 상태 파일 — HANDOFF.md · CLAUDE.md로 재개 가능

✓ Eval 게이트 — PR · Nightly · Weekly 단계별 품질 판정

✓ 격리 복구 — 한 에이전트 실패가 전체를 무너뜨리지 않음

"open" 에이전트 생태계 · 무엇을 선택할까

독립 OSS 하네스 · 리스크

버전 업데이트·호환성 관리를 팀이 짊어짐 · 관측·보안 훅을 처음부터 설계 · 모델·도구가 분리돼 Eval 신호 약함.

Claude Code + MCP · 장점

3세대 하네스 기본 요소가 내장 · 플러그인·Skills로 조직 자산 축적 · 모델·도구·하네스가 한 공급자에서 나와 디버그가 빠름.

현실적 권고
사내 프로덕션은 Claude Code + MCP를 기본선으로 두고, 오픈 하네스는 실험·리서치 트랙에서만. 두 트랙의 Golden Set과 Eval 결과를 비교해 분기마다 선택을 재검토하는 구조 권장.

PART 2 · 사내 구축 사례 3종 (35분)

커뮤니티 팀이 실제로 설계·운영한 세 가지 AI 제품의 구조를 공유합니다. 각 사례는 문제 정의 · 아키텍처 · 핵심 선택 · 실측 효과 · 러닝 포인트 다섯 단락으로 정리합니다.

2.1 사례 ① 페르소나 기반 UT 피드백 에이전트

UT(사용성 테스트)에 3~5명 정도의 실제 사용자를 매번 섭외하기 어려울 때, 페르소나를 에이전트로 구현해 1차 스크리닝을 자동화한 사례. 정성 인터뷰의 부하를 크게 줄였습니다.

목표와이어프레임 단계에서 1차 피드백을 24시간 내

핵심 선택

① 페르소나는 파일로 · 모델이 아니라 프롬프트로 · 페르소나별 파인튜닝 대신 persona.json + 시스템 프롬프트 구성. 변경·확장이 저렴.

② 출력은 "질문의 재료"로만 · AI 피드백을 결론으로 쓰지 않고, 실 UT 때 물어볼 질문 8개로 전환해 사용.

③ Golden Set · 실 UT 30건과 상관 분석 · 에이전트가 지적한 이슈가 실 UT에서 얼마나 재등장하는지 월간 비교. 상관이 낮아지면 페르소나 팩 개편.

한계와 주의

감정·맥락·문화적 뉘앙스는 여전히 실사용자만 포착 · "페르소나가 실사용자"라는 착각 방지 · 분기마다 페르소나 팩 재검수 필수.

복제하고 싶은 팀에게
페르소나 5개 중 2개는 실존 사용자 인터뷰에서 추출한 "원본"으로 시작하세요. 순수 상상 페르소나로만 구성하면 팀의 편견이 증폭됩니다. 원본 비율 40% 이상 유지가 경험칙.

2.3 사례 ③ 차량 추천 AI 에이전트

신규 서비스 기획 과정에서 직접 구축한 내 상황에 맞춘 차량 추천 에이전트. 단순 카탈로그 검색이 아니라, 사용자 라이프스타일·예산·유지비·상징성까지 고려해 2~3대를 추천하는 구조.

항목내용
기존 검색"가격·연식·세그먼트" 필터만 제공. 의사결정 부담은 사용자 몫
목표대화형 Q&A로 상황 파악 → 2~3대 맞춤 추천 + 이유 설명
제약잘못된 추천이 실제 구매로 이어지지 않도록 "최종 선택은 사람" 명시

2.4 세 사례의 공통 교훈

서로 다른 세 에이전트지만, 설계 결정이 겹치는 지점이 뚜렷합니다. 이 교집합이 사내 다음 프로젝트의 출발선입니다.

✓ "최종 결정은 사람" — 자동화의 범위와 수동 결정 지점을 UX에 명시했을 때 도입 저항이 낮아진다.

✓ Clarifier는 짧게, Explainer는 풍부하게 — 입구는 가볍게·출구는 깊게.

✓ Skill·페르소나·프리셋은 파일로 관리 — 모델을 바꾸지 않고도 조직 자산을 축적.

✓ Golden Set · 분기 리뷰 — 설계 가정이 여전히 맞는지 분기마다 재검증.

✓ "재료를 만들고 결정은 사람" — 세 사례 모두 이 슬로건으로 UX가 정리됨.

왜 지금 이 주제인가 · 팀에 주는 3가지 시사점

① 복잡한 오케스트레이션 < 단단한 루프 · Ralph의 교훈은 "과도한 설계 금지". 우리 사내 에이전트도 설계 검토 시 "더 줄일 수 있나?"를 먼저 묻자.

② 하네스는 비용이 아니라 자본이다 · Trace·Eval·훅 투자는 초반 부담이 크지만, 세 달 뒤부터 같은 팀 인원으로 두 배의 에이전트를 운영할 수 있다.
