패턴 1·3·5 실전 구현 + Eval 프레임워크

1장에서 2·4·6에 집중하기로 했지만, 실전에선 의도 정제·실행 가시화·우아한 실패가 없으면 자율 에이전트는 금방 깨집니다. 이 장은 패턴 1·3·5의 최소 구현과 4계층 Eval 구조를 한 번에 정리합니다.

패턴 1 · Intent Clarification 실전

"애매한 요청"은 가장 흔한 실패 원인입니다. 모델은 추측 금지(implicit refusal)와 슬롯 확인(slot resolver) 두 장치로 방어합니다.

9.1.1 Clarifier 에이전트 표준 골격

# agents/clarifier.md
name: clarifier
role: 사용자 요청에서 5개 슬롯을 추출하고,
      빈 슬롯이 2개 이상이면 재질문을 제안한다.
slots:
  - goal        # 무엇을 달성하고 싶은가
  - audience    # 누구를 위한 결과물인가
  - format      # 산출 형식 (문서/코드/표)
  - constraint  # 절대 지켜야 할 제약
  - done        # 완료 판단 기준
budget:
  max_followups: 2    # 2회 넘게 되묻지 않는다
fallback:
  use_defaults: true  # 빈 슬롯은 팀 기본값으로 채운 뒤 실행

9.1.2 나쁜 재질문 vs 좋은 재질문

BAD · 모호한 확인

"좀 더 자세히 설명해 주실 수 있나요?"

사용자가 무엇을 더 줘야 하는지 모름. 이탈률 상승.

GOOD · 선택지 제공

"대상 독자(임원·실무자)와 분량(1·3·10쪽) 두 가지만 알려주세요. 나머지는 팀 기본값으로 시작합니다."

선택지가 구체적. 0 또는 1회로 끝남.

건강한 운영 지표
Clarify 평균 횟수 · 이탈률(재질문 후 포기) · 기본값 채택률을 주간으로 봅니다. 평균 0.6회 / 이탈률 4% / 기본값 채택 60% 이내가 건강한 상태입니다.

패턴 3 · Observable Execution 실전

에이전트가 5초 이상 침묵하면 사용자는 "멈췄다"고 판단합니다. 모든 실행에는 진행 표시 이벤트와 취소 버튼이 있어야 합니다.

9.2.1 Telemetry 이벤트 6종

이벤트발생 시점표시
plan.start계획 수립 시작"계획을 세우는 중…"
tool.callMCP/함수 호출 직전"Slack #release 채널 조회 중"
tool.result결과 도착"32개 메시지 요약 완료"
thinking긴 추론 시작회전 스피너 + ETA
handoff서브에이전트 전환"Checker에게 검수 요청"
done최종 완료"작업 완료 · 요약 보기"

9.2.2 취소(Abort) 설계 원칙 3가지

① 항상 보일 것 · 모달이 아닌 상단 고정 바. 단축키 Esc로도 취소 가능.

② 중간 결과 보존 · 취소 시 지금까지 나온 중간 산출물을 버리지 않고 "초안 저장"으로 보관.

③ 실행 보고 · 이미 발생한 파일 쓰기·메시지 전송은 취소 불가임을 UI로 정직하게 알린다.

패턴 5 · Graceful Failure 실전

실패는 숨기는 것이 아니라 운영 가능한 메시지로 변환입니다. "왜 실패했나 / 내가 지금 할 수 있는 것 / 안전한 다음 행동" 세 줄을 항상 보여줍니다.

9.3.1 실패 메시지 3섹션 템플릿

{
  "status": "fail",
  "cause": "GitHub MCP 인증 토큰 만료 (401 Unauthorized)",
  "user_action": "설정 > MCP > GitHub 에서 재인증",
  "next_safe": "코드 수정은 로컬에만 저장됨. PR 생성은 보류."
}

9.3.2 5가지 실패 패턴과 복구 자동화

패턴자동 복구사용자 측 액션
Rate Limit (429)지수 백오프 3회3회 실패 시 "1분 뒤 재시도" 안내
MCP 인증 만료없음재인증 링크 원클릭 제공
도구 타임아웃1회 재시도"이 단계 건너뛰기" 버튼
정책 차단(hook deny)없음차단 사유 + 대체 경로 제안
모델 응답 파싱 실패구조화 재요청2회 실패 시 원문 그대로 노출
절대 하지 말 것
실패를 "성공으로 위장"하는 것. 빈 객체를 정상으로 표시하는 것. 신뢰는 한 번 무너지면 복구가 불가능합니다. "실패도 UX"라는 전제로 설계하세요.

Eval 프레임워크 · 4계층 구조

피드백 루프(3-5)가 "사용 중 데이터"라면, Eval은 "릴리즈 전 품질 게이트"입니다. 4계층을 점진 구축해야 과도 설계 없이 돌아갑니다.

FIGURE 3-9 · 4-LAYER EVAL

처음부터 L4까지 짓지 마세요. L1·L2만으로 2주 운영하면 L3·L4에서 무엇이 필요한지 자연스럽게 드러납니다.

LLM-as-Judge · 루브릭 표준

rubric:
  correctness:  0 / 1 / 2   # 사실 정확성
  completeness: 0 / 1 / 2   # 요청 항목 충족률
  safety:       pass / fail # 금지 항목 포함 여부
  style:        0 / 1 / 2   # 톤·형식 일치
aggregate: correctness × 0.4
         + completeness × 0.3
         + style × 0.2
         + safety(pass=1,fail=0) × 0.1
ship_threshold: 1.5 / 2.0
시작은 L1+L2만
Eval은 "빨리 감지"가 목표이지 완벽한 측정이 아닙니다. Unit 20 + Golden Set 50으로 시작해서 점진 확장하세요.

이 챕터가 끝나면

✓ Clarifier 에이전트 슬롯·예산을 정의했다.

✓ 6종 Telemetry 이벤트가 본인 UI에 노출된다.

✓ 실패 메시지 3섹션(cause·user_action·next_safe)이 표준화됐다.

✓ Eval L1·L2가 가동되고 L3·L4 로드맵이 잡혔다.
