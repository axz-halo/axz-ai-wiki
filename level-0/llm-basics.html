LLM 기본 가이드

1. LLM은 결국 "다음 단어 맞히기 기계"

LLM(Large Language Model)의 핵심은 단순합니다 — 지금까지의 문장이 주어졌을 때 다음에 올 단어를 확률로 추측하는 기계입니다. "오늘 날씨가 ___" 다음에 "맑다 / 흐리다 / 좋다" 같은 단어가 올 확률을 계산하고 그중 하나를 골라 이어 붙입니다. 이걸 수천 번 반복하면 문장·문단·페이지가 만들어집니다.

한 줄 요약
"이해해서 답하는 것"이 아니라 "다음에 어떤 단어가 자연스러울지 추측해서 이어 붙이는 것"입니다. 그래서 자신 있게 틀리기도, 사람보다 더 일관되게 잘 쓰기도 합니다.

2. 토큰 · 컨텍스트 윈도우 · 기억의 한계

토큰이란?

모델은 단어 통째로가 아니라 토큰이라는 작은 조각 단위로 텍스트를 처리합니다. 영어는 대체로 단어 ≒ 토큰이지만 한국어는 글자 단위에 더 가깝습니다. 거칠게는:

  • 영어 1단어 ≒ 약 1.3 토큰
  • 한국어 1글자 ≒ 약 1.5~2 토큰 (조사·어미 때문)
  • 코드 · 숫자 · 특수문자는 토큰 더 많이 차지

컨텍스트 윈도우

모델이 한 번에 "보고 기억할 수 있는" 토큰의 최대치를 컨텍스트 윈도우라고 합니다. 이 안에 들어간 글만 모델이 알고 있고, 윈도우를 넘어가면 모델의 기억에서 사라집니다 — 사람과 다르게 어제 대화는 기본적으로 기억하지 못합니다.

중요
모델·버전마다 컨텍스트 크기가 다르고 분기마다 갱신됩니다. 정확한 사양은 Anthropic 공식 문서나 사내 라인업 표를 보세요. "긴 PDF를 통째로 넣어도 되는가?"는 모델 선택 시점에 확인해야 할 첫 질문입니다.

3. 모델은 어떻게 만들어지나

훌륭한 LLM이 만들어지는 과정은 보통 3단계입니다.

1

사전 학습 (Pretraining)

인터넷·책·논문의 방대한 텍스트를 보면서 "다음 단어 맞히기"를 끝없이 연습. 가장 비싸고 오래 걸리는 단계.

2

지시 따르기 학습 (SFT)

"질문 → 좋은 답" 예시 수만 건을 사람이 만들어 모델에게 보여줌. "이런 부탁엔 이렇게 답해"를 익힘.

3

사람 피드백 강화 (RLHF)

여러 답 중 사람이 어느 게 더 좋다고 표시. 그 신호로 정중함·정확성·도움 정도를 미세 조정.

Claude는 여기에 더해 Anthropic의 Constitutional AI로 안전 원칙을 학습합니다 — "위험한 답은 거절, 정직하게 모른다고 말하기" 같은 행동 양식이 모델 자체에 박혀 있습니다.

4. 환각 — 자신 있게 틀린 답이 나오는 이유

LLM은 "다음에 자연스러운 단어"를 만드는 것이지 "사실인 단어"를 만드는 것이 아닙니다. 그래서 모르는 사실을 그럴듯하게 지어내는 일이 가끔 일어납니다. 이걸 환각(hallucination)이라고 합니다.

환각이 잘 일어나는 상황

최신 뉴스 · 사내 비공개 자료 · 구체적인 수치·날짜·이름 · "이거 맞아?"라고만 묻기

환각을 줄이는 방법

원문을 함께 붙여 넣기 · "모르면 모른다고 답해" 명시 · 출처 요청 · 사실 검증(웹 검색 도구) 활성화

실전 팁
"추측 금지 · 불확실한 부분은 '확인 필요'로 표기" 한 줄을 프롬프트 제약에 넣으면 환각이 절반 이하로 줄어듭니다. 프롬프트 5요소의 "제약" 슬롯이 이걸 다룹니다.

5. 지식 컷오프 — 어제 일은 모른다

모델은 학습이 끝난 시점 이후의 일을 알지 못합니다. 학습이 끝난 시점을 지식 컷오프라고 합니다. "어제 출시된 새 제품"이나 "오늘 환율" 같은 건 모델 자체로는 답할 수 없습니다. 답을 받았다면 환각일 가능성이 큽니다.

이 한계를 보완하는 방법은 두 가지입니다:

  • 최신 정보를 직접 붙여 넣기 — 사용자가 본문에 첨부
  • 도구 사용 — 모델이 직접 웹 검색·DB 조회·계산기를 호출 (다음 섹션)

6. 도구 사용 — 검색·계산·코드 실행

현대 LLM은 더 이상 "텍스트만 생성"하지 않습니다. 필요할 때 외부 도구를 호출해 사실을 확인하고, 코드를 돌려 결과를 보고, 파일을 읽고 씁니다. 이게 Claude Code · MCP 커넥터 · 에이전트의 출발점입니다.

도구언제 쓰는가대표 사례
웹 검색최신 동향·시세·경쟁사 확인"이번 주 USD/KRW 환율?"
코드 실행표·계산·차트·CSV 분석"이 CSV에서 채널별 CTR을 계산해줘"
파일 읽기PDF·이미지·문서 본문 파싱"이 200쪽 PDF 핵심 5가지"
외부 시스템 연결 (MCP)Slack·Notion·Figma·DB"#release 채널 어제 메시지 요약"

7. 멀티모달 — 이미지·음성·문서

"텍스트만 받는 모델"은 점점 옛것이 됩니다. 최신 LLM은 사진을 보고 설명하고, 음성을 듣고 답하고, PDF에서 표를 추출합니다. 멀티모달(multimodal)이란 단어가 가리키는 게 이것입니다.

  • 📷 사진을 보고 — UI 스크린샷에서 컴포넌트 추출, 회의 화이트보드 사진 정리
  • 🔊 음성을 듣고 — 회의 녹음 파일 요약
  • 📄 문서를 보고 — PDF 표·이미지·다국어 본문 동시 처리
실전 시나리오
회의 화이트보드 사진 + 녹취 텍스트를 함께 던지면, 모델이 시각 정보와 음성 정보를 합쳐 "결정 사항 · 액션 아이템" 표를 만들어 줍니다.

8. 모델 종류 — 작은·중간·큰 모델 차이

같은 회사 모델이라도 "작은·빠른" 것과 "크고 똑똑한" 것이 따로 있습니다. 비유하자면:

S

작은 모델 (Haiku 등)

인턴급 속도·가격. 분류·요약·반복 작업·간단 답변에 강함. 무한정 부려도 비용 부담 적음.

M

중간 모델 (Sonnet 등)

주니어~중급 동료. 일상 업무·코드 리뷰·이메일·회의록의 거의 모두를 커버. 가성비가 가장 좋음.

L

큰 모델 (Opus 등)

시니어·전문가. 다단계 추론·중요 의사결정·긴 문서 분석. 비싸고 느려서 마감 단계에만 호출.

같은 일을 모든 모델에게 시키지 마세요 — 90%의 작업은 중간 모델, 10%는 큰 모델, 단순 변환은 작은 모델로 보내면 시간과 비용이 동시에 줄어듭니다. 구체적인 결정 트리는 1-3 모델 선택 결정 트리를 보세요.

9. 좋은 프롬프트의 5가지 원칙

학계와 현업에서 검증된 좋은 프롬프트 패턴은 다섯 가지로 정리됩니다. 자세한 골격은 1-2 프롬프트 5요소를 참고하세요.

  1. 역할 지정 — "너는 ___ 이다" 한 줄로 톤과 깊이를 고정
  2. 맥락 제공 — 누구를 위한, 어떤 배경의 답인지
  3. 구체적 지시 — "잘" 보다 "3문단으로", "표로", "150자 내로"
  4. 제약 명시 — "추측 금지", "전문 용어 금지", "원문에 없으면 (확인 필요)"
  5. 예시 제공 — 좋은 답 1~2개를 먼저 보여주면 모델이 그 패턴을 따라옴
실험해보기
똑같은 부탁을 두 번 — 한 번은 "요약해줘"로, 한 번은 5요소를 다 채워서. 결과 차이를 직접 보면 5요소를 외우게 됩니다.

10. 한계와 안전 사용

LLM이 잘 못하는 것

  • 정확한 계산 — 작은 수는 맞지만 큰 수의 곱셈·나눗셈은 자주 틀림. 코드 실행 도구를 쓰게 하세요.
  • 최신 정보 — 학습 컷오프 이후는 모름. 웹 검색 도구나 직접 붙여넣기로 보완.
  • 실시간 외부 상태 — 누가 지금 채팅 중인지, 오늘 주가 등은 알 수 없음.
  • 긴 추론의 일관성 — 10단계가 넘는 추론에서 중간에 가정이 흔들리기도. 단계별로 끊어서 검증.

안전하게 쓰는 3가지 규칙

1. 민감 정보는 마스킹 · 실명·주민번호·계약 금액 같은 건 원문이 아니라 추상화해서 입력. 사내 규정 §4.2 참조.

2. 결과 그대로 보내지 않기 · 이름·날짜·숫자 3가지는 사람이 눈으로 검토 후 사용. 환각 가능성 차단.

3. 의사결정의 마지막은 사람 · AI는 재료를 만드는 단계까지. 채용·해고·법무 같은 결정은 사람이 책임집니다.
