G. 히어로 딥다이브 6선

카탈로그의 수십 개 도구 중, "이번 분기 내로 팀 워크플로우에 실제 꽂아 쓰면 효과 큰" 6개만 골라 출처·설치·주의점을 정리합니다.

H1. Superpowers (obra/superpowers) · L2~L3

푸는 문제 · "Skill·훅·서브에이전트를 한 번에 팀에 배포"하는 표준. Jesse Vincent가 Prime Radiant에서 운영하는 에이전트 스킬 프레임워크 + 소프트웨어 개발 방법론. 서브에이전트 주도 개발이 핵심 개념.

# 공식 마켓플레이스에서 설치
/plugin install superpowers@claude-plugins-official

# 대표 스킬
# - brainstorming     : 코드 쓰기 전 아이디어 정제
# - using-git-worktrees : 격리된 워크스페이스 생성
# - writing-skills    : 새 Skill을 체계적으로 작성

# 첫 작업 예시
"/brainstorm 이벤트 공지 CMS 프로토타입"

✓ 자주 넘어지는 3가지 · 자율 루프를 너무 길게 둠 · 브레인스토밍 건너뛰기 · git worktree 규칙 안 맞춤

✓ 쓰지 말아야 할 때 · "5분 걸리는 작은 수정". 설계·계획·리뷰 루프가 오히려 느려진다.

H2. Task Master AI (eyaltoledano/claude-task-master) · L2

푸는 문제 · "요구사항 한 문단을 받아도 어디서부터 쪼개야 할지 모르겠다"는 구간을 자동화. PRD를 입력으로 받아 작업 트리·의존성 그래프·마일스톤으로 쪼개줍니다.

npm install -g task-master-ai
task-master init
task-master parse-prd docs/prd.md
# → tasks/ 디렉터리에 쪼개진 작업이 생긴다

# MCP로 Claude Code에 연결 (권장)
claude mcp add task-master -- npx task-master-ai

✓ 주의 · PRD 너무 짧으면 쪼개진 작업도 뭉툭 · 의존성 무시 시 재작업 · 일단 다 돌리지 말고 "첫 마일스톤만"

✓ 쓰지 말 때 · PRD 없이 대충 시작하는 1인 PoC.

H3. Spec Kit (github/spec-kit) · L2~L3

푸는 문제 · "PRD → 기술 계획 → 테스트 가능한 작은 작업"의 표준 흐름 제공. GitHub이 2025년 오픈소스로 공개한 Spec-Driven Development 툴킷. 핵심은 constitution.md — 프로젝트의 비타협 원칙을 명문화한 문서.

speckit init ./my-project

# 생성되는 파일들
# constitution.md : 비타협 원칙 (예: "관리자 기능엔 항상 감사 로그")
# spec.md         : 제품 명세
# plan.md         : 기술 계획
# tasks/          : 테스트 가능한 작은 작업 집합

✓ constitution을 안 쓰면 매 세션마다 다른 판단. 팀 규율은 이 파일 하나로 결정.

✓ 기존 코드에 덮어씌우지 말 것 — 그린필드에 특히 강함.

✓ spec을 안 업데이트하면 코드가 spec을 다시 생성. 다음 세션에서 되돌아감.

H4. Context7 (upstash/context7) · L2~L3

푸는 문제 · "Next.js 14의 최신 API가 뭐였지?" 같이 모델 학습 시점 이후 달라진 라이브러리 문서를 Claude가 자주 헷갈리는 문제. 공식 문서를 실시간으로 가져와 Claude 컨텍스트에 주입하는 MCP.

claude mcp add --scope user context7 -- \
  npx -y @upstash/context7-mcp --api-key YOUR_API_KEY

# 프롬프트에서 트리거
"Next.js 14 서버 액션으로 폼 처리 예시 작성. use context7"

✓ "use context7" 트리거 문구가 필수. CLAUDE.md에 "문서 요청 시 항상 use context7"로 규칙화.

✓ 라이브러리 id 모호하면 resolve-library-id 먼저 호출.

✓ 비공식·내부 라이브러리는 커버 안 됨.

H5. Vanna AI (vanna-ai/vanna · MIT) · L2~L3

푸는 문제 · "자연어 → SQL"의 정확도를 조직 SQL 히스토리로 학습해 올린다. 20k+ stars, MIT 라이선스. Postgres·MySQL·Snowflake·BigQuery·Oracle·DuckDB·ClickHouse 지원. Ollama 등 로컬 LLM도 가능.

pip install vanna

# 학습 (DDL + 과거 쿼리 + 문서)
vn.train(ddl="CREATE TABLE events (...)")
vn.train(sql="SELECT COUNT(*) FROM events WHERE status='active'")
vn.train(documentation="status='active'는 오픈 이벤트만 해당")

vn.ask("이번 달 활성 이벤트 수는?")
# → SQL + 결과 DataFrame + Plotly 차트

✓ DDL만 학습시키지 말고 DDL · 과거 쿼리 · 업무 용어 설명 셋 다 먹여야 정확도 상승.

✓ 운영 DB에 직접 붙이지 말고 읽기 전용 복제본에.

✓ PII·민감 필드는 가명화 선행. 외부 LLM 사용 시 유출 위험.

H6. LangGraph (langchain-ai/langgraph) · L3

푸는 문제 · 상태·분기·재시도가 많은 업무 에이전트를 그래프로 명시 설계. "Clarifier → Retriever → Ranker → Explainer" 같은 파이프라인이 차량 추천 사례의 구조. 시각화된 상태 그래프로 "왜 이 결정이 나왔나"를 복기 가능.

pip install langgraph langchain-anthropic

from langgraph.graph import StateGraph, END
g = StateGraph(State)
g.add_node("clarify",  clarifier_fn)
g.add_node("retrieve", retriever_fn)
g.add_node("rank",     ranker_fn)
g.add_node("explain",  explainer_fn)
g.set_entry_point("clarify")
g.add_edge("clarify",  "retrieve")
g.add_edge("retrieve", "rank")
g.add_edge("rank",     "explain")
g.add_edge("explain",  END)
app = g.compile()

✓ State가 30필드 넘어가면 디버깅 지옥. 서브그래프로 쪼개기.

✓ checkpointer 없이 긴 그래프 돌리면 중간 실패 시 처음부터. SQLite 체크포인터라도 꼭 연결.

✓ 조건부 엣지가 5개 넘으면 그래프가 아니라 스파게티. Router 노드로 묶기.
