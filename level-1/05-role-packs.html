직무별 프롬프트 팩

글로벌에서 가장 많이 쓰이는 프롬프트 패턴을 5개 직무에 맞춰 조립한 팩입니다. 복사해 자신의 업무 맥락만 끼워 넣으면 오늘 당장 쓸 수 있습니다.

공통 골격 · 7대 프롬프트 패턴

아래 7개 패턴은 영어·한국어·일본어 실사용 데이터에서 공통으로 상위를 차지하는 기법입니다. 직무별 팩의 모든 예제는 이 골격 위에 얹혀 있습니다.

#패턴한 줄 설명
01Role Prompting"You are an expert ~"로 시작해 모델의 응답 톤·전문성을 고정
02Chain of Thought"단계별로 생각해" / "Think step by step"으로 추론 과정을 먼저 쓰게 함
03Few-shot Examples좋은 예 2~3개를 먼저 보여주고 "이 스타일로"
04Structured OutputJSON/마크다운/표로 명시. 후속 자동화의 기본기
05Constraint"항상 ~ / 절대 ~ 금지" 조합으로 환각·탈선 차단
06Self-Critique초안 후 "위 답을 다시 점검하고 3가지 약점을 찾아 고쳐"로 2차 개선
07Meta / Reflection"이 작업을 더 잘 하려면 내가 무엇을 더 제공해야 해?"

5.1 기획·PM 팩 — 스펙·PRD·회의

주 활용 Skill: Projects · Claude Design · Web Search. 모호한 아이디어를 구조화된 문서로 바꾸는 데 최적화된 팩입니다.

프롬프트 1 · PRD 초안 생성

[역할] 10년차 제품 기획자.
[맥락] 아래 한 줄짜리 아이디어를 PRD 초안으로 확장.
[아이디어] """ {여기에 한 줄} """
[형식]
# PRD
## 1. 문제 정의
## 2. 타깃 사용자 (페르소나 2종)
## 3. 가설 (H1, H2)
## 4. 성공 지표 (선행·후행 2개씩)
## 5. MVP 범위 (포함·제외 각 5개)
## 6. 리스크 · 오픈 이슈
[제약] 숫자는 모두 "TBD"로 비워두고 근거가 있을 때만 채워.

프롬프트 2 · 회의록 → 결정·액션 분리

[역할] 회의록 편집자.
[지시] 아래 녹취를 "결정·액션·이슈" 3섹션으로 분리.
[형식]
## 결정 (주제 · 결정 · 근거)
## 액션 (담당 · 내용 · 마감)
## 미결 이슈 (질문 · 누가 답해야 하는가)
[제약] 녹취에 없는 내용은 추측 금지.

프롬프트 3 · 경쟁 제품 벤치마크

[역할] 제품 리서처. Web Search 활용 가능.
[지시] 아래 3개 제품을 "핵심 기능 · 가격 · 타깃 · 차별점" 4열 표로.
[제약] 공식 웹/가격 페이지만 소스로. 블로그 루머 금지.
금지 패턴
"좋은 PRD 써줘" 같은 모호한 단일 문장 프롬프트는 금지. 숫자·페르소나·성공 지표의 포맷을 항상 함께 지정해 초안 품질을 강제하세요.

5.2 디자이너 팩 — UX 카피·리뷰·목업

주 활용 Skill: Claude Design · Artifacts · Projects. 시각 작업의 앞뒤에 붙는 텍스트·검토 작업을 자동화하는 팩입니다.

프롬프트 1 · UX 카피 3안 생성

[역할] 프로덕트 카피라이터.
[맥락] {화면 설명}. 사용자 상태: {초심·숙련}.
[지시] 버튼·에러·빈 상태 UX 카피를 각각 3개씩 제안.
[형식]
## 버튼 (3개) - 직설/친절/짧은
## 에러 (3개) ...
## 빈 상태 (3개) ...
[제약] 한국어. 각 문구 18자 이내. 느낌표·물음표 금지.

프롬프트 2 · 디자인 리뷰 체크리스트

[역할] 시니어 디자이너 리뷰어.
[지시] 첨부 시안을 아래 기준으로 점검. 각 항목 OK·주의·수정.
[체크]
- 정보 계층 (타이포 크기 3단 이상 분리)
- 클릭 가능 영역 (44×44 이상)
- 색 대비 (WCAG AA)
- 에러·성공 상태 커버
- 로딩·빈 상태 커버
[형식] 5×3 표(항목·상태·코멘트).

프롬프트 3 · Claude Design 목업 시드

[역할] Claude Design 디자인 에이전트.
[지시] 아래 서비스의 핵심 화면 3종 와이어프레임 생성.
[화면] 온보딩 · 메인 홈 · 설정
[형식] 각 화면마다 레이아웃 + 섹션 요소 리스트 + 컴포넌트 후보 3개
[제약] 사내 디자인 시스템 토큰만 사용.

5.3 개발자 팩 — 디버깅·리뷰·리팩터

주 활용 Skill: Artifacts · Claude Code · Analysis. 초급 개발자도 시니어 옆자리 페어링 효과를 얻는 팩입니다.

프롬프트 1 · 에러 메시지 → 3분 디버깅

[역할] 시니어 엔지니어. 단계별로 생각해.
[지시] 아래 에러의 근본 원인을 "가설 3개 → 검증 방법 → 수정안" 순서로.
[형식]
## 가설 (순위순)
1. ... (왜 그렇게 생각하는지)
## 검증
- 각 가설별로 "터미널에서 이걸 실행"
## 수정안
- diff 형태로
[제약] 임의로 라이브러리를 바꾸지 말고, 먼저 현재 코드로 해결 시도.

프롬프트 2 · PR 자가 리뷰

[역할] 코드 리뷰어. 본인이 작성한 PR을 엄격히 점검.
[지시] 아래 diff에서 다음 6가지 관점으로 문제 찾기.
1) 의도 불명 이름
2) 테스트 누락 경로
3) 에러 경로 누락
4) 로그·메트릭 부족
5) 성능 리스크
6) 보안/권한
[형식] 각 카테고리별 "파일:줄 · 지적 · 대안 코드" 표.

프롬프트 3 · 테스트 케이스 생성

[역할] QA 엔지니어.
[지시] 아래 함수의 단위 테스트를 생성.
[포함]
- Happy path 2개
- 경계값 3개 (null, 빈 배열, 최대치)
- 실패 경로 2개
- 시간대 · 로케일 이슈 1개
[형식] pytest 스타일. 각 테스트는 주석으로 "무엇을 검증하는지" 1줄.
금지 패턴
회사 비밀 코드 전체를 붙여 넣지 마세요. 함수 한 개·50줄 이내로 맥락만 남기고 요청. API 키·DB 접속 문자열은 반드시 삭제.

5.4 마케터 팩 — 카피·분석·리서치

주 활용 Skill: Web Search · Analysis · Artifacts. 카피 생성과 데이터 해석을 한 세션 안에 묶는 팩입니다.

프롬프트 1 · 멀티채널 카피 10안

[역할] B2C 그로스 마케터. 타깃: {세그먼트 설명}.
[지시] 아래 캠페인 오퍼를 4개 채널(이메일 제목·인스타·틱톡·검색광고)
       각각 톤을 달리해 10안씩 생성.
[형식] 4채널 × 10개 표. 각 카피 옆에 "훅 키워드" 1개.
[제약] 한국어. 이메일 제목 30자 이내, 인스타 120자 이내.

프롬프트 2 · 캠페인 성과 데이터 분석

[역할] 마케팅 애널리스트. Analysis(파이썬) 활용.
[지시] 첨부 CSV를 읽고 다음을 계산·시각화.
1) 채널별 CTR · CVR · CPA
2) 전주 대비 WoW 변화
3) 이상치 (평균 ±2σ 밖)
[형식] 표 + 막대그래프 + "오늘 할 일 3가지" 실행 요약.
[제약] 숫자는 소수점 2자리. 통화 단위 ₩ 유지.

프롬프트 3 · 경쟁 캠페인 조사

[역할] 시장 조사원. Web Search 활용.
[지시] 아래 3개 경쟁사의 최근 30일 캠페인을 요약.
[수집]
- 캠페인 메시지 · 채널 · 추정 예산 레벨
- 소비자 반응 (리뷰·언급 톤)
[형식] 3×4 표 + "우리가 따라갈 것/피할 것" 요약 2줄.
[제약] 모든 사실 옆에 출처 URL 병기.

5.5 HR·인사 팩 — 채용·온보딩·피드백

주 활용 Skill: Projects · Web Search · Artifacts. 휴먼 터치가 핵심인 영역에서 초안 부담만 덜어주는 팩입니다.

프롬프트 1 · JD 초안

[역할] 10년차 HR 파트너.
[지시] 아래 요건으로 JD 초안을 작성.
[입력]
- 직무: {예: 시니어 프론트엔드}
- 팀: {설명}
- 필수 3 / 우대 3
[형식]
## 역할 요약 (3문장)
## 주요 업무 (5~7개 불릿)
## 필수 · 우대 (구분해서)
## 우리가 제공하는 것 (문화·성장)
[제약] 성별·나이·학력 특정 표현 금지. 한국어 존대.

프롬프트 2 · 면접 질문 세트 20개

[역할] 시니어 면접관. 행동 기반 면접(BEI) 전문.
[지시] 아래 JD에 맞춘 질문 20개.
[분배]
- 역량 검증 10개 (STAR 기반)
- 문화 적합 5개
- 케이스 · 실무 5개
[형식] 표: 번호 · 질문 · 평가 포인트 · 주의(유도 금지 표현).

프롬프트 3 · 건설적 피드백 리포트

[역할] 매니저. 성과 리뷰 초안 작성.
[지시] 아래 사실 관찰을 "Situation · Behavior · Impact · Suggestion"
       구조로 재작성.
[형식]
## 잘한 점 (SBI 2개)
## 개선할 점 (SBI 2개 + 제안)
## 다음 분기 목표 (측정 가능 3개)
[제약] 성격·인성 표현 금지. 관찰 가능한 행동만.
민감 정보
이 팩의 모든 프롬프트는 개인 식별 정보(PII)를 제거한 익명 데이터로만 사용하세요. 실명·주민번호·급여는 절대 프롬프트에 넣지 않습니다.

이 챕터가 끝나면

✓ 내 직무에 해당하는 팩에서 즉시 쓸 수 있는 프롬프트 3개를 골랐다.

✓ Notion Prompt Kit에 본인용 팩을 시작했다.

✓ 직무별 금지 패턴(예: 코드 통째 붙여넣기, 모호한 PRD 요청)을 안다.
