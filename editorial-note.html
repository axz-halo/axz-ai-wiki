편찬자 노트

이 위키 시리즈를 편찬한 커뮤니티 팀 헤일로의 운영·자산 관점 노트입니다. 세 가지 본책과 세 가지 별책을 만들면서 부딪힌 벽, 그 벽을 넘어 가는 방법, 그리고 매일의 작업 습관 — 이 페이지는 본책의 "정리된 결론"과는 다른, 날 것 그대로의 운영 노트입니다.

1. Claude는 "결과 생성기"가 아니라 "공동 편집자"

헤일로는 Claude에게 "써줘"라고 부탁하지 않습니다. "같이 정리해줘"라고 부탁합니다. 결과 품질은 사람과 Claude의 협업에서 나온다는 전제 위에 서 있습니다.

BAD · 결과 의뢰형

"PRD 써줘" "와이어프레임 만들어줘"
→ 첫 결과를 받고 통째로 폐기하거나 사람이 다시 씀.

GOOD · 공동 편집형

"A/B/C 안을 같이 뽑아보자" + "B안의 컴포넌트와 C안의 톤을 합쳐줘"
→ 사람은 결정·근거·취향만, Claude는 안 생성·정리·반복 수정 담당.

실전 적용
매 프롬프트 시작 시 "내가 결정자, Claude는 편집자"라는 역할을 정해 두면 의존도 폭주가 없어집니다. 결정은 사람이, 반복은 Claude가.

2. 한 번에 완성 말고 "이것도 부탁해"로 점진 확장

이 위키도 처음부터 23챕터를 한 번에 만들지 않았습니다. v1.0(샘플 챕터 3개) → v1.1(L1 완성) → v1.2(L2 완성) → v1.3(L3 완성) → v1.4(실습 트랙 + 카탈로그) 순서로 점진 확장했습니다.

점진 확장의 사이클

① 한 번 만든다 (v1.0) · 완벽하지 않아도 끝까지 간다. "완성 후 평가"가 "평가 후 완성"보다 항상 빠르다.

② Claude에게 갭 감사를 시킨다 (v1.1) · "이게 90점 이상일까?" "무엇이 빠졌을까?" — 인간 리뷰어보다 빠르고 편견 없다.

③ 갭을 채우고 밀도를 조절한다 (v1.2~) · 타이포·밀도·톤 같은 감각 영역을 여러 번 미세 조정. "읽는 감각"이 산출물 품질을 결정한다.

결정타
한 번에 다 만들려는 사람이 제일 늦게 끝낸다. 빨리 만들고 자주 다듬는 사람이 결국 가장 많이 쌓는다.

3. 기초 공정 → 심화 공정 · 파이프라인으로 쪼개기

헤일로의 사내 기획 작업은 Skill 한 개로 끝나지 않습니다. 두 개의 Skill을 파이프라인으로 잇는 방식입니다.

단계Skill 이름 · 하는 일
기초 공정
(문제·배경 정리)		 halo-basic-service-plan · AS-IS 분석 · 경쟁사 벤치마크 · 유저 플로우 · 앱+웹 화면 스펙 · 로드맵. 산출물은 Markdown + PDF.
심화 공정
(스펙·와이어프레임)		 halo-wireframe-webapp · 기초 공정 산출물을 입력받아 인터랙티브 와이어프레임 + 6관점 엘리먼트 + SlotSpec · OperationalSpec으로 확장.
왜 두 개로 쪼갰는가
하나의 거대한 Skill은 유지 보수가 어렵고, 기능 추가 시 다른 부분을 깨뜨리기 쉽습니다. "입력·출력이 명확한 작은 Skill을 파이프로 연결"하면 각각을 독립적으로 개선할 수 있습니다. 초급 별책의 "점진 확장" 원칙을 Skill 설계에 적용한 버전입니다.

4. 자기 방법론을 Skill로 "화석화"하기

헤일로는 본인의 기획 프로세스 — 문제 기반 프레이밍 → 정책 판단 → 운영 스펙(데이터 파이프라인 · 캐싱 · refresh 정책) → 엣지 케이스 매트릭스 → AS-IS→TO-BE 변경 추적 — 를 SKILL.md에 박제했습니다.

반복 작업 소멸
같은 포맷 10번째 쓸 때 Claude가 이미 포맷을 안다
팀 전수 가능
신입이 첫 주에 선임 수준 결과물
자기 성장 가시화
분기별 개정을 git log로 확인
전환의 의미
개인 역량을 Skill로 박제하는 순간, "대체 불가능한 장인"에서 "팀 전체를 레벨업시키는 설계자"로 역할이 바뀝니다. 이게 시니어가 Skill을 만드는 진짜 이유입니다.

5. 감각적 튜닝 · 여러 버전 돌리기

이 위키 시리즈도 v1.0 → v1.4까지 여러 버전을 돌리며 만들어졌습니다. 프로토타입 품질의 결정타는 "얼마나 많은 버전을 돌리느냐"입니다.

매 버전 점검하는 감각 체크리스트

✓ 본문 밀도 — 문단이 답답하지 않은가. 줄간격·자간은 오래 읽어도 피로하지 않은가.

✓ 톤 일관성 — 1장과 5장의 문체가 이질적이지 않은가.

✓ 도표·표·콜아웃의 비율 — 글만으로 지루한 구간이 3페이지 이상 이어지지 않는가.

✓ 공지·약속의 일치 — 앞에서 "다룬다"고 한 것이 실제로 뒷장에 있는가.

중급이 바로 쓰는 법
자기 Skill에도 같은 원칙을 적용하세요. v1.0을 먼저 만들고, Claude에게 "이 Skill이 놓치고 있는 케이스 3가지를 말해줘"를 시키세요. Skill이 진화하는 속도가 팀 내 영향력의 성장 속도입니다.

7. 타임리스/이벤트 분리 · 트렌드 큐레이션

위키 콘텐츠를 두 종류로 분리해 관리합니다. 타임리스는 본질 콘텐츠 — 5요소·결정 트리·5섹션 HANDOFF.md처럼 기술이 바뀌어도 유효한 원칙. 이벤트는 시점 콘텐츠 — Opus 4.7 출시, 특정 라이브러리 동향, 분기별 사례.

구분예시갱신 주기
타임리스5요소 · 결정 트리 · HANDOFF.md 포맷 · 4계층 Eval · 30항 체크리스트분기 1회 검토
이벤트모델 라인업·도구 카탈로그·사내 사례 수치월 1회 갱신
트렌드 큐레이션 루틴
매주 금요일 30분 — Awesome Claude Code · MAGI//ARCHIVE · Anthropic Engineering Blog를 훑고 새 도구 1개를 도구 카탈로그에 추가. 1년이면 50개 도구가 자연스레 쌓입니다.

끝 · 72시간 원칙

위키를 만들었으면 사용되어야 합니다. 헤일로가 매번 적용하는 72시간 원칙입니다.

✓ 출시 72시간 안에 동료 3명이 위키의 한 챕터를 실제 업무에 적용했다.

✓ 1주 안에 Notion Prompt Kit에 본인 직무 템플릿 1개를 저장했다.

✓ 2주 안에 같은 패턴으로 두 번째 적용을 만들었다. 두 번째가 더 빠르면 체화 완료.

✓ 분기 안에 본인이 만든 Skill 1개를 사내 레포에 커밋했다.

이 위키 시리즈와의 연결

지금까지 만든 본책 3권(초급·중급·고급)과 별책 3권(세미나 실습)은 모두 한 가지 목표를 향합니다 — AI를 도구에서 동료로 바꾸기.

이 위키는 그 여정의 표준화된 입구입니다. 동료들이 같은 출발선에 서고, 같은 어휘로 일하며, 같은 안전망 위에서 실험하기 위한 공동 기준점.

초급의 반복은 중급에서 자동화로, 중급의 자동화는 고급에서 제안으로 진화합니다. 각 단계의 주인공은 언제나 여러분입니다.
